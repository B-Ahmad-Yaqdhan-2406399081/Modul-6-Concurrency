# Commit 1 Reflection
In the `handle_connection` method, we are passing a mutable `TcpStream` which represents the connection between the client and the server. We wrap this stream in a `BufReader` to efficiently read the incoming byte stream line by line. The `http_request` variable is created by reading lines from the `buf_reader`, mapping the `Result` to its underlying `String` using `unwrap()`. Furthermore, we are taking lines until an empty line is encountered, which signifies the end of the HTTP request headers. Finally, we collect these parsed lines into a `Vec<_>` (a vector of strings) and print it. This simple mechanism allows us to inspect the raw HTTP request sent by the browser. We can clearly see the HTTP method (like GET), the path (like /), the HTTP version, and various request headers.

# Commit 2 Reflection
![Commit 2 screen capture](./assets/images/commit2.png)

In this commit, the `handle_connection` function is enhanced to actually respond back to the client instead of just printing the request. We define a `status_line` variable containing "HTTP/1.1 200 OK" to indicate a successful HTTP response. Next, we read the contents of the `hello.html` file into a string using `fs::read_to_string`. To comply with the strict HTTP protocol, we must explicitly calculate the `Content-Length` by getting the byte length of our HTML string. We then format the complete HTTP response by combining the status line, the Content-Length header, and the HTML body. A crucial part of this formatting is the double carriage return and line feed (`\r\n\r\n`) used to clearly separate the headers from the main body. This fully formatted string is converted to raw bytes and written back to the `TcpStream` using `stream.write_all()`, effectively sending the webpage to the user's browser.

# Commit 3 Reflection
![Commit 3 screen capture](./assets/images/commit3.png)

To make our web server more robust and useful, we implemented request validation to distinguish between requests for the root path (`/`) and other unknown paths. By inspecting the `request_line` (the very first line of the HTTP request), we can accurately determine what the client is asking for. We can then selectively respond with the `hello.html` file for successful root requests and a `404.html` file for anything else. Refactoring the code using an `if-else` or `match` statement is highly necessary to avoid duplicating the file reading and response writing logic for every single route. By extracting the status line and filename into variables determined by the matching logic, we significantly reduce code duplication. This refactoring makes the code much cleaner and easier to maintain. It also prepares the fundamental structure for adding more complex routing in the future without cluttering the function with repetitive boilerplate code.

# Commit 4 Reflection
By adding a `/sleep` endpoint that triggers a `thread::sleep(Duration::from_secs(10))`, we deliberately simulate a heavy or slow operation on our server. Because our server is currently single-threaded, it processes requests strictly sequentially, one after another. When a user requests the `/sleep` path, the entire main thread of the server pauses for a full 10 seconds. During this blocked period, if another user tries to access the normal root path (`/`), their browser will just hang and wait continuously. This happens because the server is completely blocked and fundamentally unable to accept or process new incoming TCP connections. This experiment clearly demonstrates the severe limitation of a single-threaded server in a concurrent environment, as one slow request can degrade the experience for all other users simultaneously. It highlights the urgent need to implement a multithreaded architecture to handle concurrent connections independently and efficiently.

# Commit 5 Reflection
To overcome the severe blocking issue of our single-threaded server, we implemented a robust `ThreadPool` architecture. A thread pool essentially creates a specific, fixed number of worker threads upfront when the server starts, and these threads wait for incoming tasks to execute. When our server receives a new connection, instead of handling it directly on the main thread, it sends the connection task into the pool. The thread pool then assigns this task to an available worker thread via a channel, specifically using `mpsc` (multiple producer, single consumer). If all threads are currently busy processing requests, the new task simply waits in the channel's queue. Once an idle thread becomes available, it picks up the next task from the queue and executes it concurrently. This scalable architecture ensures that a slow request (like our `/sleep` endpoint) only occupies one single worker thread, allowing the other threads in the pool to continue serving new incoming requests without any noticeable delay.
